# 3티어 아키텍처에 맞는 Security Groups 규칙 설계하기

3티어 아키텍처는 애플리케이션을 **웹 서버**(프레젠테이션 계층), **앱 서버**(비즈니스 로직 계층), **DB 서버**(데이터 계층) 세 역할로 나누는 구성입니다. 사용자 요청을 웹 서버가 받아 앱 서버로 전달하고, 앱 서버가 DB 서버에서 데이터를 조회해 응답을 반환하는 흐름입니다. 쇼핑몰, 사내 업무 시스템, API 기반 서비스 등 대부분의 웹 서비스가 이 구조를 기반으로 합니다.

NHN Cloud에서 티어를 별도 인스턴스로 분리하면 다음과 같은 이점이 있습니다.

- **독립 확장**: 트래픽이 늘어난 티어만 인스턴스를 추가해 확장할 수 있습니다. 예를 들어 웹 서버에 요청이 몰리면 웹 서버 인스턴스만 늘리면 됩니다.
- **장애 격리**: 앱 서버에 장애가 발생해도 DB 서버는 영향을 받지 않으므로 데이터 유실 위험이 줄어듭니다.
- **보안 강화**: 외부 사용자가 접근하는 웹 서버와 데이터가 저장된 DB 서버 사이에 앱 서버가 위치하므로, 외부에서 DB 서버로 직접 접근할 수 없습니다.

이 가이드에서는 세 번째 이점인 보안 강화를 NHN Cloud Security Groups로 구현하는 방법을 다룹니다. 티어를 분리하더라도 모든 포트를 열어 두면 한 티어가 침해되었을 때 나머지 티어까지 노출됩니다. 보안 그룹(Security Groups)은 인스턴스의 인바운드·아웃바운드 트래픽을 규칙 단위로 제어하는 가상 방화벽입니다. 이 가이드는 티어별로 보안 그룹을 분리하고, 각 티어가 최소 권한 원칙에 따라 필요한 트래픽만 허용하도록 규칙을 설계하는 시나리오를 다룹니다.

## 시작하기 전에

- NHN Cloud 프로젝트에 VPC(Virtual Private Cloud)와 서브넷(Subnet)이 구성되어 있어야 합니다.
- 각 티어에서 사용할 포트와 프로토콜을 파악합니다.
    - 예: 웹 서버 80/443, 앱 서버 8080, DB 서버 3306
- NHN Cloud 콘솔에 로그인할 수 있는 계정이 필요합니다. NHN Cloud 계정 생성 방법은 [계정 생성과 로그인](https://docs.nhncloud.com/ko/quickstarts/ko/create-account/)을 참고하세요.

## 트래픽 흐름 설계하기

규칙을 만들기 전에, 어떤 트래픽이 어디에서 어디로 흘러야 하는지 먼저 정리합니다. 트래픽 흐름을 매트릭스로 정리하면 불필요한 규칙을 만들지 않고, 누락된 규칙도 쉽게 발견할 수 있습니다.

다음은 3티어 아키텍처의 트래픽 흐름 매트릭스 예시입니다.

| 출발지 | 도착지 | 프로토콜 | 포트 | 용도 |
|---|---|---|---|---|
| 인터넷(0.0.0.0/0) | 웹 서버 | TCP | 80, 443 | HTTP/HTTPS 접속 |
| 관리자 IP | 웹 서버 | TCP | 22 | SSH 관리 접속 |
| 웹 서버 | 앱 서버 | TCP | 8080 | 앱 API 호출 |
| 앱 서버 | DB 서버 | TCP | 3306 | MySQL 쿼리 |

!!! tip "알아두기"
    NHN Cloud Security Groups는 stateful 방화벽입니다. 규칙으로 한 번 연결된 세션은 반대 방향의 규칙이 없더라도 허용됩니다. 예를 들어 인바운드 TCP 80 규칙으로 들어온 트래픽의 응답은 별도 아웃바운드 규칙 없이도 자동으로 허용됩니다. 단, 일정 시간 패킷이 들어오지 않아 세션이 만료되면 반대 방향 패킷도 차단됩니다.

## 보안 그룹 만들기

티어별로 보안 그룹을 분리하면 각 역할에 맞는 규칙만 관리할 수 있고, 나중에 인스턴스를 추가하거나 교체할 때도 보안 그룹만 연결하면 됩니다.

### 콘솔에서 설정하기

다음 단계를 따라 3개의 보안 그룹(sg-web, sg-app, sg-db)을 만듭니다.

1. NHN Cloud 콘솔에서 **Network > Security Groups**를 클릭하세요.

2. **보안 그룹 생성**을 클릭하세요.

3. 웹 서버용 보안 그룹 정보를 입력합니다.
    - **이름**: `sg-web`
    - **설명**: `웹 서버 티어 - HTTP/HTTPS 허용`

   이름은 영문자, 숫자, `-`, `_`만 사용할 수 있으며 255자 이내로 입력합니다. 티어 역할이 드러나는 이름을 사용하면 나중에 규칙을 관리할 때 어느 보안 그룹이 어떤 역할인지 바로 알 수 있습니다.

4. 하단의 보안 규칙은 이 단계에서 추가하지 않고 **확인**을 클릭합니다. 규칙은 다음 단계에서 설계에 맞게 추가합니다.

5. 같은 방법으로 다음 두 보안 그룹을 추가로 만듭니다.

   | 이름 | 설명 |
   |---|---|
   | `sg-app` | `앱 서버 티어 - 웹 서버에서만 접근 허용` |
   | `sg-db` | `DB 서버 티어 - 앱 서버에서만 접근 허용` |

## 인바운드 규칙 추가하기

인바운드 규칙은 외부에서 인스턴스로 들어오는 트래픽 중 어떤 것을 허용할지 정의합니다. 최소 권한 원칙에 따라 각 티어가 실제로 필요로 하는 트래픽만 허용합니다.

### 콘솔에서 설정하기

#### sg-web: 웹 서버 인바운드 규칙

1. 보안 그룹 목록에서 `sg-web`을 클릭하세요.

2. **보안 규칙** 탭에서 **보안 규칙 생성**을 클릭하세요.

3. 다음과 같이 HTTP 수신 규칙을 입력합니다.

   | 필드 | 값 | 설명 |
   |---|---|---|
   | **방향** | 수신 | 외부에서 들어오는 트래픽 |
   | **IP 프로토콜** | HTTP | 포트 80 자동 지정 |
   | **Ether** | IPv4 | |
   | **원격** | CIDR — `0.0.0.0/0` | 모든 IP에서 접근 허용 |
   | **설명** | `HTTP 접근 허용` | |

4. **확인**을 클릭한 뒤, 같은 방법으로 다음 규칙들을 추가합니다.

   | 방향 | IP 프로토콜 | Ether | 원격 | 설명 |
   |---|---|---|---|---|
   | 수신 | HTTPS | IPv4 | CIDR — `0.0.0.0/0` | HTTPS 접근 허용 |
   | 수신 | SSH | IPv4 | CIDR — `{관리자 IP}/32` | SSH 관리 접속(관리자 IP만) |

!!! danger "주의"
    SSH 원격 값에 `0.0.0.0/0`을 입력하면 모든 IP에서 SSH 접속이 가능해집니다. 반드시 관리자 IP만 허용하세요. 콘솔의 **원격** 드롭다운에서 **내 IP**를 선택하면 현재 접속 중인 IP가 자동으로 입력됩니다.

#### sg-app: 앱 서버 인바운드 규칙

1. 보안 그룹 목록에서 `sg-app`을 클릭하세요.

2. **보안 규칙** 탭에서 **보안 규칙 생성**을 클릭하세요.

3. 다음과 같이 입력합니다.

   | 필드 | 값 | 설명 |
   |---|---|---|
   | **방향** | 수신 | |
   | **IP 프로토콜** | 사용자 정의 TCP | |
   | **포트** | `8080` | 앱 서버 포트 |
   | **Ether** | IPv4 | |
   | **원격** | 보안 그룹 — `sg-web` | 웹 서버 보안 그룹에서만 접근 허용 |
   | **설명** | `웹 서버에서 앱 API 호출 허용` | |

   **원격**에서 CIDR 대신 **보안 그룹**을 선택하면 해당 보안 그룹에 속한 인스턴스의 IP가 변경되더라도 규칙이 자동으로 따라갑니다. IP 주소를 직접 관리할 필요가 없어 운영 부담이 줄어듭니다.

4. **확인**을 클릭합니다.

#### sg-db: DB 서버 인바운드 규칙

1. 보안 그룹 목록에서 `sg-db`를 클릭하세요.

2. **보안 규칙** 탭에서 **보안 규칙 생성**을 클릭하세요.

3. 다음과 같이 입력합니다.

   | 필드 | 값 | 설명 |
   |---|---|---|
   | **방향** | 수신 | |
   | **IP 프로토콜** | MY SQL | 포트 3306 자동 지정 |
   | **Ether** | IPv4 | |
   | **원격** | 보안 그룹 — `sg-app` | 앱 서버 보안 그룹에서만 접근 허용 |
   | **설명** | `앱 서버에서 MySQL 접근 허용` | |

4. **확인**을 클릭합니다.

## 아웃바운드 규칙 설정하기

보안 그룹을 새로 생성하면 기본 아웃바운드 규칙으로 모든 트래픽 송신이 허용됩니다(송신/임의/IPv4 `0.0.0.0/0`, 송신/임의/IPv6 `::/0`). 3티어 아키텍처에서는 이 기본 규칙을 유지하는 것이 일반적입니다.

!!! tip "알아두기"
    아웃바운드를 더 엄격하게 제어해야 하는 환경(예: 금융, 의료 등 컴플라이언스 요구사항이 있는 경우)이라면 기본 송신 규칙을 삭제하고, 필요한 대상만 명시적으로 허용할 수 있습니다. 이 경우 패키지 업데이트(80/443), DNS 조회(53) 등 운영에 필수적인 아웃바운드도 별도로 허용해야 합니다.


## 동작 확인하기

규칙을 모두 적용한 뒤, 각 티어 인스턴스에 보안 그룹을 연결하고 트래픽이 의도한 대로 흐르는지 확인합니다.

1. 각 인스턴스에 해당 보안 그룹을 연결합니다.
   - 웹 서버 인스턴스 → `sg-web`
   - 앱 서버 인스턴스 → `sg-app`
   - DB 서버 인스턴스 → `sg-db`

   보안 그룹을 연결하려면 **Compute > Instance**에서 인스턴스를 클릭한 뒤 **네트워크** 탭으로 이동합니다. 네트워크 인터페이스를 선택하고 **보안 그룹 변경**을 클릭하면 체크박스로 보안 그룹을 선택하거나 해제할 수 있습니다.

2. 웹 서버에서 외부 접근을 확인합니다.

   ```sh
   # 로컬 PC에서 웹 서버 Floating IP로 접속
   curl -I http://{웹 서버 Floating IP}
   # HTTP 200 응답이 오면 정상
   ```

3. 웹 서버에서 앱 서버로 내부 통신을 확인합니다.

   ```sh
   # 웹 서버 인스턴스에 SSH 접속한 뒤 실행
   curl http://{앱 서버 사설 IP}:8080/health
   # 앱 서버 응답이 오면 정상
   ```

4. 앱 서버에서 DB 서버로 내부 통신을 확인합니다.

   ```sh
   # 앱 서버 인스턴스에서 실행
   mysql -h {DB 서버 사설 IP} -u {사용자} -p
   # MySQL 접속이 되면 정상
   ```

5. 차단이 의도한 대로 동작하는지 확인합니다. 다음 접속은 실패해야 합니다.

   ```sh
   # 웹 서버에서 DB 서버로 직접 접속 시도 → 차단되어야 함
   mysql -h {DB 서버 사설 IP} -u {사용자} -p
   # 타임아웃 또는 연결 거부 발생 시 정상
   ```

   웹 서버에서 DB 서버로 직접 연결이 차단되는 것은 `sg-db`의 인바운드 규칙이 `sg-app`만 허용하기 때문입니다.

## 응용하기

- **인스턴스 추가**: 같은 티어에 인스턴스를 추가할 때는 해당 보안 그룹만 연결하면 됩니다. 규칙을 다시 만들 필요가 없습니다. 예를 들어 웹 서버를 2대로 늘린다면 새 인스턴스에 `sg-web`을 연결하기만 하면 동일한 규칙이 적용됩니다.
- **포트 변경**: 앱 서버 포트를 8080에서 다른 포트로 변경한다면, `sg-app`의 인바운드 규칙에서 포트 값만 수정합니다. 보안 그룹 참조 방식이므로 `sg-web` 쪽은 수정할 필요가 없습니다.
- **4티어 이상 확장**: 캐시 서버나 메시지 큐 같은 티어가 추가되는 경우에도 같은 패턴을 적용합니다. 새 보안 그룹(예: `sg-cache`)을 만들고, 해당 티어에 접근이 필요한 보안 그룹만 원격으로 지정합니다.

## 용어 정리

| 용어 | 설명 |
|---|---|
| 보안 그룹 | 인스턴스의 출입 트래픽을 제어하는 것 |
| VPC | 가상 사설 클라우드. 논리적으로 분리된 가상의 네트워크 |
| 서브넷 | Subnetwork를 의미하여, IP 네트워크의 더 작은 범위로 세분화된 IP 주소 범위 |
| CIDR | 클래스 없는 도메인 간 라우팅의 약어로서, IP 주소를 효율적으로 할당하기 위한 IP 주소 할당 방식 |
| 인스턴스 | 가상의 CPU, 메모리, 기본 디스크로 만들어진 가상 서버 |
| 방화벽 | 미리 정의된 보안 규칙에 따라 들어오는 네트워크 트래픽을 모니터링하고 제어하는 네트워크 보안 시스템 |
