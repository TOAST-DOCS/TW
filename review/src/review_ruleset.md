## 듀얼 렌즈
- **ⓐ 변경분(`grade_change`)**: 6월 배포 diff(추가/변경된 부분)만 검수. 사후 검수 본래 취지.
- **ⓑ 전체(`grade_full`)**: 문서 전체 검수(기존 이슈 포함).
- 한 번에 검수하되 각 지적을 출처 태깅: `[변경분]`(diff에서 발생) / `[기존]`(원래 있던 이슈) / `[참고]`(오탐 가드·맥락 메모, 수정 대상 아님).

## 등급 기준
- 🔴 **R**: 기술 오류 / 링크 깨짐(외부 `http://`·죽은 AWS 링크 등) / 필수 섹션 누락 / 콘솔 UI 불일치
- 🟠 **O**: 용어·글로서리·조사·어미·서식 문제
- 🟢 **G**: 통과(고칠 것 없음 또는 사소)
- ⓐ/ⓑ 각각 R/O/G 부여. 변경분이 깨끗하면 ⓐ=G, 기존 이슈만 있으면 ⓑ만 O/R.

## 용어·글로서리 (위반 시 🟠)
- `Console` → `콘솔`
- `그룹태그` → `그룹 태그`
- `NHN Cloud 회원` → `NHN Cloud 계정`
- `IAM 멤버` → `IAM 계정` (+조사: 멤버를→계정을, 멤버로→계정으로, 멤버는→계정은)
- UI 경로 `고객 센터 > 1:1 문의` → `고객지원 > 문의하기`
- `글로사리` → `글로서리`
- 버킷: `버킷명` 금지 → `버킷 이름`(띄어쓰기)
- 서비스 상태: stop=`정지`, terminate=`종료`. `중지`는 사용 안 함(예외: RDS for MySQL DB 인스턴스만 '중지' 허용).
- **용어를 바꾸면 뒤따르는 조사도 반드시 함께 점검**(와/과, 를/을, 로/으로, 는/은, 가/이): 받침 유무 변화 확인.

## 띄어쓰기 (위반 시 🟠)
- 붙여 쓰기: `미리보기`, `키페어`, `비즈니스폼`, `메타스토어`, `상세 보기`는 띄움(주의: '상세 보기'는 띄움)
- `-명`은 앞말에 붙임(Instance명, 조직명, 그룹명) / `- 이름`은 앞말과 띄움(Instance 이름)
- `제한 값`은 띄움
- 단위는 숫자에 붙임: `100GB`, `1,000`(천 단위 콤마)
- **`-명` ↔ `이름` 상호 변환은 일괄 적용 금지**(건별 판단). `조직명`은 그대로 유지.

## 어미·표현 (위반 시 🟠)
- `참고하시기 바랍니다` / `참고합니다` → `참고하세요`로 통일
- 링크 마무리 표현(`~에서 확인할 수 있습니다` ↔ `~항목을 참고합니다` 등)은 **원문 그대로 유지**(통일 강제 금지)
- 번역투(피동 남용, 부사 '보다'=비교 의미 아닌 경우) 점검
- 만연체(한 문장 여러 줄) → 문장 분리(1문장 1메시지)

## 서식 (위반 시 🟠)
- 점 목록(`-`)은 **여러 항목 나열 시에만**. 단일 문장은 평문.
- 불릿 기호 `*`/`+` → `-`
- 노트 박스(MkDocs admonition): 알아두기=`!!! tip`, 주의=`!!! danger`, 본문 4칸 들여쓰기. `>` 인용은 예시 블록 전용.
- 본문/코드에 이모티콘(`^^`, `ㅋㅋ` 등) 금지.
- 특수문자 점검: EN DASH(U+2013)·EM DASH(U+2014)·NBSP(U+00A0)가 하이픈/공백 자리에 잘못 쓰였는지 확인.

## 릴리스 노트 전용 규칙 (release-notes.md / release-note.md / release-notes-*.md)
- 구조: H1=제목 / H2=날짜 / H3=카테고리.
- 부모 불릿=명사형, 자식 불릿=`-됩니다`체.
- 수정·명시 완료 항목은 과거형. `1,000` 천 단위 콤마. '동일' 선호.

## 오탐 가드 (절대 수정 지적하지 말 것 — `[참고]`로만 메모)
- **통신사(SKT/KT/LGU+) 외부 고객센터** 전화번호·명칭에 `고객 센터→고객지원` UI 룰 적용 금지.
- **서비스명을 직접 가리키는 문맥**이면 콘솔 정착 영문 유지(글로서리 한글화 무시).
- 알림 수신 그룹의 `회원`·`NHN Cloud 비회원`은 별개 개념 → `계정` 변환 금지.
- 오류 코드 표/메시지의 `고객문의로 문의하세요` 등은 원문 보존(UI 경로 룰은 본문에만).
- Certificate Manager 문맥은 `알림`(`알람` 아님).
- 콘솔 메시지의 정착 표현(`진행하다`·`이용하다`·`~으로 인해`·`존재하여`)은 유지 경향(조사·어미·글로서리만 수정).
- **기술적 사실이 달라지는 변경**(종료/재시작/정지 등 동작 의미 변화)은 임의 수정 지적 금지 → `[참고]`로 "작성자 확인 필요"만.

## 출력 파일 형식 (반드시 준수 — 빌드 스크립트가 파싱함)
파일 경로: `reviews/<service>__<문서키>.md`
 - 문서키 = path에서 `ko/` 제거하고 `/`를 `__`로, `.md` 제거. 예: `ko/console-guide/template.md` → `console-guide__template`.
 - 예: notification-hub `ko/console-guide/template.md` → `reviews/notification-hub__console-guide__template.md`

frontmatter + 본문:
```
---
service: <서비스 repo명>
path: <ko/...md 전체경로>
status: <A 또는 M>
grade_change: <G|O|R>
grade_full: <G|O|R>
review_state: auto
---

# <service> / <path> — 자동 검수 결과

> Claude 자동 검수 초안입니다. 담당 라이터가 항목을 검토·반영(`[x]` 체크)하고,
> 완료되면 상단 `review_state`를 `done`(검토 중이면 `reviewing`)으로 바꿔 주세요.

## 변경해야 할 항목
- [ ] [기존] <행번호> <지적 내용 — 무엇을 어떻게 고칠지>
- [ ] [변경분] <6월 변경 중 문제가 된 부분>
- [x] [변경분] <6월 변경이 적절해 조치 불필요한 경우 — 체크해서 기록(대시보드엔 숨겨짐)>
- [ ] [참고] <오탐 가드·맥락 메모>

## 참고
- 등급: ⓐ 변경분 = <통과/수정 필요/시급> / ⓑ 문서 전체 = <...>
- <요약 한두 줄>
```

규칙:
- `[ ]`=미조치(수정 필요), `[x]`=완료/조치 불필요. 대시보드 '변경해야 할 항목'에는 `[ ]`만 노출됨.
- 백틱 코드는 `` `...` `` 로 표기(빌드가 <code>로 렌더).
- 행번호는 원문(content) 기준으로 명시하면 라이터가 원본 대조하기 쉬움.
- 신규(A) 문서는 변경분=전체이므로 ⓐ=ⓑ로 동일하게 판정.

### 항목 작성 스타일 (반드시 — 대시보드는 한눈에 스캔되어야 함)
- **1 항목 = 1 이슈 = 한 줄.** 형식: `[태그] <행번호> <무엇> → <어떻게>`. 한 줄(최대 1~2문장)로 끝낼 것.
- **추론·근거·장문 설명을 항목 줄에 넣지 말 것.** 부연이 꼭 필요하면 `## 참고` 섹션으로 내릴 것.
- **여러 이슈를 한 항목에 합치지 말 것.** 이슈마다 별도 `- [ ]` 줄로 분리.
- **조치할 수 없는 관찰("작성자 판단", "의도된 표기일 수 있음", "정상임")은 `[ ]` 액션 항목으로 올리지 말 것.** → 판단이 필요하면 `[참고]` 태그로, 검수 결과 문제없으면 `[x]`로(또는 아예 생략).
- **"작성자 확인 필요"(기술적 사실·기능 의미 변화)는 `[참고]` 태그**로 한 줄.
- 좋은 예: `- [ ] [기존] 631행 \`[Andorid]\` 오타 → \`[Android]\``
- 나쁜 예: 한 항목에 명사형/종결형/혼재/렌더링까지 다섯 가지를 6줄로 서술.
